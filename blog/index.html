Anne でブログを書くことにした

なんか。 いろいろあって Anne というシリアライズフォーマット兼軽量マークアップ言語でブログを書くことにした。

今日はその話。

背景

もともとわたしはひとより自然言語の文章をよく書くタイプだと思う。 ひとより、というのは、一般的なプログラマに比べて。

そういうのもあり、文章を書くのは非常にたいへんだと理解している。 文章自体を書くのもたいへんだし、それを書くフォーマットに振りまわされるのもつらい。

ドキュメントやブログもそうなのだが、文章を書くという行為において 人間はふたつのものと戦わねばならない。 ひとつは英語や日本語のような自然言語。もうひとつは Word だとか LaTeX だとか、 自然言語を入力するソフトウェア。

当然のことだが、もし LaTeX を完璧に使いこなせたとしても、自然言語の部分を書く能力や労力は別途求められる。 その逆もまた然りで、自然言語で完璧に文章を書けたとしても、 LaTeX が使えるとはかぎらない。

このふたつはべつの行為であり、しかし、コンピュータで文章を書く以上、同時にこなさなければならないことでもある。

そしてやはりもちろん、両方を同時にこなすのは難しい。 自然言語を書きつつ、ソフトウェアのチューニングもするのは困難なことだ。

テーリとオセロできるだけの VR ゲームがつくりたいという話、あと近況

ゲーム開発は難しいなぁという話。

ゲーム開発、何度か挑戦してるんだけどうまくいかない。じゃあなにがダメだったの？ とかの反省をしつつ、これからどうしようねみたいな。

なにがダメだったかと言えば、正直なところ、 最初から壮大なものをつくろうとしすぎていたところに終始する気がする。 いや、そんなのみんな言ってるしわかりきってたことでしょ、 というのはそうなんだけど。

でも、最初から壮大なものをつくろう、みたいな意識はまったくなかったというか、 むしろ簡単なものをつくろうみたいな感じで要件定義していたはずなんだよね。 それでもダメだった。じゃあどうして？と言えば、原因はおそらく以下のふたつ:

  • 開発したいゲームの規模を見積もる能力が足りていなかったこと。
  • 小規模でおもしろいゲームを思いつく能力がなかったこと。

前者は要するに見積もりの話。 「こういうゲームをつくりたい」というのはあっても、 そのゲームをつくるのがどれくらいたいへんなのか見積もることができない。 こっちはこっちでも問題なんだけど、もっと大きい問題は後者。

後者は「限られたリソースで、いかにおもしろいものをつくるか」という能力。 当然のことなんだけど、人間のリソースはかぎられている。 でも、おもしろさみたいなものは投資したリソースと単純に比例するわけじゃない。 （よくある炎上爆死プロジェクトみたいなのは、おもしろさみたいなものを吟味せず、 損切りできずにだらだらつまらないものにリソースを投入し続けた結果起こるんじゃないかと思っている。）

「限られたリソースで、いかにおもしろいものをつくるか」というのは言うだけなら簡単なんだけど、 実際にするのは難しいと思う。

最近 (n 週遅れだけど) Raspberry Pi をいじって遊んでいる。 いろいろいじってて、 raspi を使えばけっこうお手軽に自作 OS を実機で動かせて楽しいのでは？ と思い至ったのでその周辺のことを最近調べている。

Raspbian Install Battle

Effective Makefile Programming

Makefile は実用的なプログラミング言語である。 ここでは Makefile をもちいた実用プログラミング技法（？）を提案しよう。

Introduction

趣味・仕事問わず、プログラマならしばしば簡単なスクリプトを書く必要に迫られることがあると思う。 それは往々にしてプロダクションコードとは別の開発用の小さなツールである。 そういったツールを書くとき、どのような言語を選択するのか？ シェルスクリプト, Python, あるいはひとによっては、そのような場合にこそ Haskell を使うかもしれない。 わたしにとっては、その選択肢のひとつに Makefile がある。 これは半分は冗談だし、半分はマジメな話である。

辞書や配列といった構造を使う必要があるなら、シェルスクリプトより Python のほうが明らかに楽である。 シェルスクリプトは外部コマンドを呼びやすいというメリットがあり、 ただ単にいくつかのコマンドをまとめる程度のほんとうに単純なスクリプトの場合、 Python よりシェルスクリプトのほうが平易になりやすいと思う。

ただシェルスクリプトでどうしてもつらい場面がひとつあり、それはオプションやサブコマンドを扱いたい場面である。 辞書や配列を使うほど複雑なことはしないが、 必要に応じてパラメタライズしたいという場面はよくあり、 その場合にシェルスクリプトにはそのような機能が致命的に欠けている。

つまり基本的にはシェルスクリプトを書きたいのだが、 部分的にパラメタライズやサブコマンドといった機能を使いたい場合、 じつは Makefile の機能は非常にスクリプト言語として便利なのである。

なにはともあれ Hello world である。

.PHONY: main
main:
	@echo hello world

Cubase を購入し、 DTM を始めた。 まだほとんどループや効果音といった素材を鳴らしているだけなものの、 ゲームに音がついたことで、劇的にクオリティがあがったと思う。

ドローンをだしいれできるようになった

いままでドローンはハイジャックすると単に敵味方が反転するだけで、性能や位置などは変わらなかった。 基本的にゲームバランスの関係上ドローンの移動速度はテーリより遅くせざるを得ず (1/3 程度) 、 ハイジャックしても同じ性能というゲームデザイン上のポリシーがあって、 テーリが移動してもついてきてくれない、などの問題があった。

そこでいったんハイジャックしたドローンはアイテム扱いで収納できるようにして、 必要に応じてだしいれできる、という処理を実装した。

このことで副次的に、地上を移動するドローン（や、まだ構想の段階ではあるが、水中でのみ使用できるドローン） などの扱いも容易になった。テーリがジャンプで高いところへ行っても、アイテムとして収納することで 持ち運びできるからだ。

レベルデザイン: ハイジャックをゲームクリアの必須要素に

全体的にレベルデザインを見直して、初期状態のテーリのジャンプ力では到達できないが、 ハイジャックしたドローンを足場にすれば先に進める、というマップを用意した。

これによってハイジャックの立ち位置が大きく変わった。 いままでは「ハイジャックを使わなくてもクリアできる」というのがレベルデザインの原則だったのに対し、 「ハイジャックを使わなければ先に進めない場所がある」というふうになったからだ。

穴に落ちたらステージの最初に戻される処理

いままでステージに穴という概念がなかったので、レベルデザインの幅が狭かった。 穴に落ちたらダメージを受けてステージの最初に戻されるようになったことで レベルデザインの幅が非常に広がった。

背景: ゲーム用の設定としてのハイジャック

テーリというキャラクターは、もともとスパイやハッカーというわけではありませんでした。 なんでもそつなくこなす万能キャラというイメージはあり情報技術についても不得手なわけではありませんでしたが、 それが本業だということになったのは、現在制作中の Hijack のゲームシステムが考えられ、 その主人公として彼女が抜擢されたあとのことです。

つまり歴史的にはハイジャックというシステムとテーリというキャラクターは独立して存在し、 Hijack の開発が始まったから彼女がハイジャックするキャラクターになったのだ、 ということになります。

なので、ハイジャックはどちらかといえばゲームシステムにとって都合のいい設定になっています。 そのためそれをもとにした SS などでは、少々アンリアルな描写にもなってしまいますが、 ご愛嬌です。

ホロイオ

ホロイオはテーリが両腕につけているブレスレットで、 Holo I/O の略です。 Holo I/O で動作するプログラムは C 言語で holoio.h を include することで開発することができるようです。
